# Code-Repo Discipline

The standard **repository shape** — the files, folders, and task-runner recipes a conforming code repository carries, regardless of language. One skeleton across all repos means an agent (or the user) landing in any project already knows where the instructions live and what `just test` does.

- **Standard key files** — every repo carries the same small set of top-level files (README, CLAUDE.md, justfile, build config); their names are load-bearing and quoted verbatim by tooling and sibling specs.
- **Standard recipe set** — the `justfile` exposes a uniform recipe vocabulary (`setup`, `build`, `test`, `lint`, `check`, `docs`, …); each language maps the names onto its own toolchain rather than inventing new ones.
- **Generated output is gitignored** — the `site/` docs-build folder is never committed; it is regenerated and deployed (e.g. to GitHub Pages).
- **Cross-project only** — an entry belongs in this discipline only if it is expected in *most* code repositories (or is a named optional with clear semantics). Project-specific recipes, layouts, and build steps live in that project's own `CLAUDE.md` / `justfile` / README.

This is a discipline, not a user-invocable skill — language-flavor disciplines ([[DSC rust]]) and repo-scaffolding skills cite it via `[[DSC code-repo]]`.

## Key repository files

- **README.md** — brief project description, installation, quick start.
- **CLAUDE.md** — Claude Code project instructions; coding conventions, key patterns.
- **justfile** — task runner entry point; the standard recipes below. Use [just](https://github.com/casey/just).
- **Build config** — the language's dependency/build manifest at the repo root (`pyproject.toml` for Python, `Cargo.toml` for Rust, …).
- **mkdocs.yml** — MkDocs documentation site configuration (optional; only when the repo builds a docs site).

## site/ folder

Generated by the documentation build. Gitignored, never committed; deployed to GitHub Pages (or equivalent) as part of the docs pipeline.

## Justfile standard

The `justfile` is the single entry point for project tasks. Recipe names are the cross-language contract — a new language flavor maps them, it does not rename them. If a new standard recipe is genuinely needed, add it here first, then mirror it in the language disciplines.

### Standard recipes

- **setup** — fix up the dev environment so all symlinks and dependencies are correct.
- **build** — perform an incremental build.
- **rebuild** — clean build from scratch.
- **test** — run the basic test suite.
- **testall** — run all maintained tests.
- **lint** — run linters (ruff, mypy, clippy, …).
- **check** — run all checks (lint + test).
- **docs** — build documentation.
- **install** — install dependencies.
- **dev** — install in development mode.
- **clean** — remove build artifacts.

### App-specific recipes

Add as needed for applications:

- **run** — run the application.
- **rerun** — stop the running app, rebuild, then launch.
- **buildrun** — build then run (without stopping first).
