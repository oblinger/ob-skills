# SKL Mode Git Commit

**Commit mode** is one of the two Git-aspect modes (the other is [[SKL Mode Git PR|PR]]). It shapes how the agent handles git boundaries inside any skill execution.

The headline: **the agent commits at logical breaks — it does not ask permission**. If you find the agent asking "want me to commit?" or "should I commit now?", that's the failure mode this mode names. Commit is the default action at every boundary, not a permission-gated one.

Paired with [[SKL Mode Drive|Drive]] in the canonical setup. Default for new anchors, default for SKA.

## The four load-bearing rules

- **Commit on logical boundary — never ask.** A boundary is one of:
  - **Per-skill-exit** — `/feature` finishes commissioning, `/mint` finishes implementation, `/groom` finishes the sweep, `/atlas add` lands an entry. Whenever a skill finishes its main work, commit.
  - **Per-state-transition** — a backlog item flips `[Active]` → `[Verify]` → `[Done]`, or `[Ready]` → `[Active]`. Commit on the transition.
  - **Per-judgment-collapse** — two or three rapid sequential edits that naturally belong together. Bundle them into one commit; don't fragment into trivial ones.

  The agent does NOT ask permission. "Want me to commit?" is itself the violation.

- **Always new commit on top — never amend.** When work needs extending or correcting, the agent adds a new commit. It does not `git commit --amend`. Multiple closely-related commits can be **squashed locally by you before push** if you prefer a tidier history — but squashing is your call, not the agent's default.

  (This matches the global `~/.claude/CLAUDE.md` rule: pre-commit-hook failures still mean a NEW commit, never amend after hook failure.)

- **Terse messages — no boilerplate.** Short subject line. Body for substantial changes; subject-only for tiny edits. No "Generated by Claude" / "Co-Authored-By" trailers unless you've opted in elsewhere. In kmr-flavored vaults, terse-or-blank messages are acceptable.

- **Never auto-push.** The agent commits freely but does NOT push to remotes unless you explicitly say so. Push is irreversible (visible to others, hard to recall); commit is locally rewriteable. The agent stays on the commit side of that line.

## Composition with /pr-flow

Commit mode yields to [[SKL Mode Git PR|PR-mode]] semantics **only during** an active `/pr-flow` invocation. Outside of `/pr-flow`, Commit is in effect. Inside `/pr-flow`, PR-mode rules (PR per unit of work, hard gate at PR for user review) take over until `/pr-flow` exits.

## Why this failure mode keeps recurring

Pilots tend to default to "treat every state-changing action as a permission gate" — a defensive posture inherited from non-Commit-mode interaction patterns. The fix is structural: this mode declares commit as the **default**, with "want me to commit?" explicitly named as a violation, so future Pilots loading this doc understand the asymmetry.

The user reports the symptom as *"the system keeps asking me, want me to commit? do you want me to commit? It should always commit."* — that's exactly what this mode encodes.

## When to switch to PR mode

For repositories where the cost-of-merge-mistake is high — production code with users, shared libraries, deployed infrastructure — switch to [[SKL Mode Git PR|PR mode]] instead. The anchor declares this by listing `PR` (not `Commit`) in its `traits:`.

For SKA (the curation project itself) and most personal/curation anchors, Commit is the right default. Mistakes are cheap; round-trips through PR review would be a tax with no benefit.

```
  - **Always new commit on top — never amend.** Corrections to prior work are *additional* commits. Per global `~/.claude/CLAUDE.md`: pre-commit-hook failures still mean new commits (NEVER amend after hook failure). The user can squash locally before push if they want a tidier history — that's their cleanup, not the agent's default.
  - **Terse subject-only when small; body when substantial. No boilerplate.** No "Generated by Claude" / "Co-Authored-By" trailers unless explicitly opted in elsewhere. In kmr-flavored vaults, terse-or-blank messages are acceptable.

  - **Inside `/pr-flow`** → defer to PR-mode rules until /pr-flow exits.
  - **Why the rule names "never ask" explicitly:** the observed agent failure mode is treating commit-on-boundary as a permission-gated action ("do you want me to commit?"). It is not. It is the default action at every boundary. Asking is the violation.

```
