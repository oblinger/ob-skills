# Finalize Discipline

The reusable ceremony for closing out a unit of work — typically a feature, but applicable to any deliverable that has its own doc, status field, and acceptance bar (a milestone, a release, a major refactor). Not directly user-invocable. Consumer skills delegate to this discipline when their workflow reaches a "this thing is done — wrap it up" moment.

The point is: closing out is *not* "commit and call it done." It's a sequence of small ceremonies that catch the things that get forgotten — failing tests pushed unnoticed, status not updated so the user can't find what was shipped, module docs that drift from the implementation. The discipline exists so each of those steps lands consistently across every consumer skill.


## Why this exists — the problem it solves

Skills like `/land`, `/crank`, and `/code release` all hit the same moment: a unit of work is done and needs to be wrapped up before the agent moves on (or stops). Without a shared discipline, each skill reinvents the wrap-up sequence — and inevitably each one forgets a different step. One commits without pushing, another pushes without updating Status, a third skips the test verification.

Centralizing the ceremony as a discipline means:
- Every consumer follows the same closing sequence.
- A new consumer (e.g., a future `/ship`) picks up the discipline by reference.
- A change to the ceremony (e.g., adding a "regenerate TOC" step) propagates to every consumer automatically.


## When to apply

A consumer skill invokes this discipline when:

- A feature has been minted and reaches its end-of-lifecycle ceremony (`/land` finishing a feature; `/crank` finishing a feature before continuing).
- A milestone or release point is reached.
- A deliverable has been verified and the agent is about to either stop or pick up new work.

Don't apply this discipline for:
- Mid-implementation commits (use ordinary commit discipline).
- Closing a backlog item that was never a "feature" (e.g., a one-off fix where nothing has a Status field to update).


## Runbook

The closing sequence — execute in order. Steps that don't apply (no feature doc, no module docs, etc.) are skipped, but the order of the rest is preserved.

### 1. Verify

- Run the relevant test suite for the work being closed. **All tests must pass** before any other step.
- If tests fail, do not proceed — fix the failures first, or escalate to the user if the cause is unclear.
- Check `git status`: nothing should be left unstaged that belongs to this work.
- Check for untracked files that should be committed (new source, new docs, new tests).

### 2. Commit

- Stage the relevant files by **specific path** (not `git add -A` — that risks pulling in unrelated noise like editor cruft or HookAnchor churn).
- Commit with a brief message: what changed and why, no boilerplate, no "generated by Claude" lines.
- Push to the tracking remote if the branch has one.
- Merge any open PRs related to the work being closed (after CI is green).

### 3. Update Feature Doc (if applicable)

If the work-unit has a dated feature doc:
- Set the `## Status` section to **Done — finalized YYYY-MM-DD**.
- Resolve any remaining `## Open Questions` entries — none should be in a pending state at finalize time. If pending questions exist, the work isn't actually done; halt and surface to the user.
- Add a brief "What shipped" summary if the implementation diverged meaningfully from the design.

### 4. Update Status / Stat Records

- If `skl-stat` is in use, post a Done update with the work-unit's S-number and a brief activity note.
- If the work corresponds to a backlog item (`F<n>` or `B<n>` in `{NAME} Backlog.md`), move it to `## Done` via the workflow skill's `backlog-edit.py` — never edit the backlog file directly. F/B-number, title, and body are preserved:

  ```bash
  ~/.claude/skills/workflow/scripts/backlog-edit.py {NAME} Done <row-id> Done
  ```

  Add the PR / commit / feature-doc cross-reference by passing it as the body arg in a follow-up call:

  ```bash
  ~/.claude/skills/workflow/scripts/backlog-edit.py {NAME} same <row-id> Done "" "Shipped 2026-MM-DD — see [[F<n> — Title]] / PR #N / commit <sha>"
  ```

  (`""` as the title preserves the existing title via the script's preserve-on-omit semantics — only the body changes.)
- If a roadmap milestone was closed by this work, mark it complete in the roadmap doc.

### 5. Update Docs to Match Reality

- If the implementation changed any **public API**: update the corresponding module doc (per `[[CAB Module Doc]]`).
- If new files were added: update `{NAME} Files.md` and the `{NAME} Dev` dispatch table.
- If architecture changed: update `{NAME} System Design.md` or `{NAME} Architecture.md`.
- If user-facing behavior changed: update `{NAME} User Guide.md`.

### 6. Clean Up

- Delete temporary or feature branches whose work has merged.
- Close related issues if they're now satisfied.
- Verify `git status` is clean — no dangling unstaged or untracked entries from this work.
- If a dev server / build process exists for the project, verify it still works.

### 7. Q.md update post-condition — automatic via `backlog-edit.py`

Step 4's `backlog-edit.py` Done call auto-refreshes the anchor's per-anchor section in `~/ob/kmr/Q.md` (by shelling out to `audit-q.py --scope backlog --anchor {NAME} --fix`). **The backlog file is NOT reordered** — source order is preserved (per F075 Q2). Bubble-to-top is a Q.md-only behavior; the user reading Q.md sees the just-finalized anchor at the top.

The audit's fix-by-default behavior catches any drift introduced — broken links, stale brackets, banner mismatches, stale `[Done]` rows — and either repairs them mechanically OR (rare) files a `QFix [Ready]` backlog entry. **Surfacing any QFix entry is part of this skill's "done" criteria** — read the `backlog-edit.py` output for QFix lines and surface them to the user.

### 8. Report

The consumer skill is responsible for reporting; the discipline supplies what to include:
- Work-unit name and identifier (feature name + S-number; or backlog B-number; or milestone name).
- Files changed (count or list, depending on consumer's verbosity).
- Tests passing (yes/no — should always be yes by step 1).
- Docs updated (which ones, briefly).

Each consumer formats this report in its own style; the discipline just guarantees the substance is complete.


## Consumers

| Skill | When it invokes this discipline |
|---|---|
| **`/land`** | When an in-flight activity is a feature/milestone wrap-up. Land iterates through every in-flight item; for each that's a finalizable unit, it runs this discipline before moving on. |
| **`/crank` / `'`** | When the agent finishes a feature mid-stream and is about to pick up the next item. The discipline runs as the close-out before the next-item selection. |
| **future `/code release` / `/ship` / similar** | Any skill that needs end-of-lifecycle ceremony for a release or deploy. |


## Anti-patterns

- **Skipping verification** — committing without running tests because "they passed an hour ago." Run them at finalize time.
- **Half-finalizing** — committing the code but not updating Status, or updating Status but not regenerating the docs. The discipline is one ceremony; partial application leaves the system inconsistent.
- **`git add -A`** — risks pulling in editor cruft, HookAnchor regenerations, build artifacts. Always stage by explicit path.
- **Boilerplate commit messages** — "Generated by Claude" / "Co-Authored-By: Claude" lines clutter the log. The user has a global rule against these.
- **Finalizing with pending questions** — if `## Open Questions` still has pending entries, the work isn't done. Surface to the user and halt rather than fabricate a Done status.
- **Auto-resuming after finalize** — finalize is a *closing* ceremony. The consumer skill decides whether to continue (`/crank`) or stop (`/land`). The discipline itself doesn't pick the next thing.
